# Data Model

The full Phase 1 Prisma schema lives in `backend/prisma/schema.prisma`. This document explains the rules around it — what the schema encodes, what the schema doesn't, and the conventions that must hold.

## Rule 1: Money is BigInt

Every monetary column is `BigInt @default(0)`. The unit is **ngwe** (1/100 of a Zambian Kwacha). K1,020.00 = `102000n` ngwe.

```prisma
contribution_amount_ngwe BigInt @default(0)
late_penalty_ngwe        BigInt @default(0)
```

- DB type: `BIGINT`
- TS type: `bigint`
- Never coerce to `Number` for arithmetic — only for display
- Format helper: `formatNgwe(amount)` (in `backend/src/lib/money.ts`, planned for Week 3)

## Rule 2: Phone numbers are E.164

Every phone column is `String` and stores E.164 only: `+260977123456`. Validation happens at the API boundary in `lib/phone.ts`; the DB doesn't enforce format, but the service layer always normalises before insert.

## Rule 3: UUIDs everywhere

Every primary key is `String @id @default(uuid())`. No auto-incrementing integers. This avoids collisions across environments and makes it safe to expose IDs in URLs.

## Rule 4: Timestamps are UTC

All `DateTime` columns use Prisma's `@default(now())` and `@updatedAt`. Prisma stores them as `TIMESTAMPTZ` in Postgres, which is always UTC. We never call `new Date()` for storage without explicit conversion.

## Rule 5: Enums for finite sets

Anything with a closed set of values is an enum: `UserStatus`, `OtpPurpose`, `InvoiceStatus`, `GroupStatus`, `GroupMemberRole`, `GroupMemberStatus`, `CycleStatus`, `RoundStatus`, `ContributionStatus`, `NotificationStatus`, `WhatsappLogStatus`. Adding a new value to a real-world enum requires a new Prisma migration.

## Rule 6: Group isolation via composite keys

Most group-scoped tables have a composite index on `(groupId, ...)` to make the dominant access pattern (list all rows for one group) fast:

```prisma
@@index([groupId, status])
@@index([groupId, createdAt])
```

## Tables

### Auth & users

| Table | Purpose |
|---|---|
| `users` | One row per real human. Owns phone, email, password hash, OTP-verified flag, role. |
| `otps` | Time-limited verification codes. Always `used = false` until verified. |

### Billing (Week 3+)

| Table | Purpose |
|---|---|
| `invoices` | SaaS billing rows. `status` = `pending` → `paid` or `cancelled`. |
| `payment_proofs` | POP uploads linked to invoices. `status` = `pending` → `approved` / `rejected`. |

### Groups

| Table | Purpose |
|---|---|
| `groups` | A Chilimba group. `invoiceId` is `UNIQUE` — every group has exactly one invoice. |
| `group_settings` | The rule book for the group. 1:1 with `groups`. |
| `group_members` | A user joined to a group with a role and payout position. `@@unique([groupId, userId])`. |

### Cycles & contributions

| Table | Purpose |
|---|---|
| `cycles` | A savings cycle within a group. `@@unique([groupId, cycleNumber])`. |
| `cycle_rounds` | The individual rounds inside a cycle, with `dueDate`. |
| `cycle_payouts` | Who got paid in a round, and how much. |
| `contributions` | Every contribution event (pending / paid / late / waived). |

### Notifications

| Table | Purpose |
|---|---|
| `notifications` | One row per outbound message intent (welcome, otp, reminder, …). |
| `whatsapp_logs` | One row per Evolution API call attempt (with response and status). |

## What's NOT in Phase 1

- `penalties` (Phase 2)
- `subscriptions` (Phase 2)
- `loans`, `loan_sureties`, `loan_repayments` (Phase 3)

These tables will be added via new Prisma migrations when their respective modules land.

## Migrations

- One migration per feature. Filename is `YYYYMMDDHHMMSS_<name>/migration.sql` (auto-generated by Prisma).
- Never edit a migration after it's been applied to a shared environment. Add a new one.
- For destructive changes: add the new column, dual-write for a release, then drop the old column in a later migration.
